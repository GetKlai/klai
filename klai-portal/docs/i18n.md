# Internationalisation (i18n)

klai-portal supports Dutch (`nl`) and English (`en`). The system is built on [Paraglide JS](https://inlang.com/m/gerre34r/library-inlang-paraglideJs).

## Structure

```
frontend/
  messages/
    en.json          # English message keys (source of truth)
    nl.json          # Dutch translations
  src/
    paraglide/
      messages.ts    # Auto-generated: exports m.key_name() functions
      runtime.ts     # getLocale(), setLocale(), availableLanguageTags
      registry.ts    # datetime, plural helpers
    lib/
      locale.ts      # LocaleContext, LocaleProvider, useLocale() hook
```

## Adding a new message

1. Add the key to `en.json`:
   ```json
   "my_new_key": "My message text"
   ```

2. Add the Dutch translation to `nl.json`:
   ```json
   "my_new_key": "Mijn berichttekst"
   ```

3. Run the Paraglide compiler (or `npm run build` — it runs automatically):
   ```bash
   npm run build
   ```

4. Use in a component:
   ```tsx
   import * as m from '@/paraglide/messages'
   <p>{m.my_new_key()}</p>
   ```

## Parameterised messages

Use `{param}` placeholders in the message value:

```json
"greeting": "Hello, {name}!"
```

```tsx
m.greeting({ name: "Alice" })  // → "Hello, Alice!"
```

## Plural forms

Use the `plural` helper from `@/paraglide/registry`:

```json
"count_one": "1 item",
"count_other": "{count} items"
```

```tsx
import { plural } from '@/paraglide/registry'
import { getLocale } from '@/paraglide/runtime'

plural(getLocale(), count) === 'one'
  ? m.count_one()
  : m.count_other({ count: String(count) })
```

## Date formatting

```tsx
import { datetime } from '@/paraglide/registry'
import { getLocale } from '@/paraglide/runtime'

datetime(getLocale(), isoString, {
  day: 'numeric',
  month: 'short',
  year: 'numeric',
})
```

## Locale switching

The active locale is managed by `LocaleProvider` (wraps the whole app in `main.tsx`). Components read and switch locale via `useLocale()`:

```tsx
import { useLocale } from '@/lib/locale'

function LocaleSwitcher() {
  const { locale, switchLocale } = useLocale()
  return (
    <>
      <button onClick={() => switchLocale('nl')}>NL</button>
      <button onClick={() => switchLocale('en')}>EN</button>
    </>
  )
}
```

Auth pages get a locale switcher via `AuthPageLayout`:

```tsx
<AuthPageLayout leftContent={...} showLocale>
  {/* content */}
</AuthPageLayout>
```

## Key naming conventions

Keys follow a `page_section_element` pattern:

| Prefix | Usage |
|--------|-------|
| `login_` | Login page |
| `forgot_` | Forgot password page |
| `set_` | Set password page |
| `signup_` | Signup page |
| `verify_` | Email verification page |
| `totp_` | TOTP login step |
| `setup_2fa_` | 2FA setup page |
| `app_` | App (chat/tools) pages |
| `admin_` | Admin area |
| `admin_billing_` | Admin billing section |
| `admin_users_` | Admin users section |
| `sidebar_` | Sidebar navigation |
| `provisioning_` | Workspace provisioning screen |
| `error_` | Shared error messages (e.g. `error_connection`) |

## Shared error keys

Connection errors that appear in multiple pages use a single shared key:

```json
"error_connection": "No connection, check your internet"
```

Do not add page-specific `*_error_connection` keys — reuse `m.error_connection()`.

Page-specific error keys that have different content (e.g. `admin_billing_error_connection`, `signup_error_connection`, `verify_error_connection`) remain separate because their copy differs.
